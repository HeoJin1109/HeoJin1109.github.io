<li>조회 성능 개선의 한계로 인해, 정합성 관리 비용을 감수하고 역정규화 방식 채택</li>
</ul>
</li>
</ul>

<blockquote>
<p>커넥션 획득 대기로 응답 지연이 누적되던 구조를 해소하기 위해 적정 Connection Pool 산정 기준을 도입한 과정</p>

<h4 id="문제-상황-1">문제 상황</h4>
<h5 id="grafana-대시보드-주요-지표">Grafana 대시보드 주요 지표</h5>

<li>
<p>500 OPS 고부하 환경에서 최대 175건의 <code>Pending Connection</code> 발생 및 처리량 저하 현상 확인</p>
</li>
<li>
<p>I/O 바운드 작업 비중이 높은 서비스 특성에 비해 커넥션 풀 크기가 작아, 요청량 대비 커넥션 공급이 부족한 상태</p>

</li>
<li>
<p>전체 응답 지연의 대부분이 DB 처리 시간이 아닌 커넥션 획득 대기에서 발생하는 구조 확인</p>
</li>
</ul>

<p>반복 검색 연산으로 자동완성 응답이 저하되던 구조를 개선하기 위해 Caffeine·Redis 기반 다계층 캐시를 도입한 과정</p>

<h4 id="문제-상황-2">문제 상황</h4>
<ul>
<li>

</li>
<li>
<p>MongoDB Atlas Search 기반 연산이 요청마다 수행되어 트래픽 증가 시 비용 증가 및 병목 발생 구조</p>

</ul>
</li>
<li>
<p>기존 DB 직접 조회 방식 사용 시 평균 응답 속도 856 ms 기록 및 UX 저해 확인</p>
</li>
</ul>
<h4 id="개선-내용-2">개선 내용</h4>
<ul>

</li>
<li>
<p>Local Cache (Caffeine)</p>
<ul>
<li>애플리케이션 메모리 내 직접 참조를 통한 네트워크 RTT 제거 및 저지연 응답 처리</li>
<li>데이터 연산 비용 최소화 및 즉각적인 응답 속도 확보</li>
</ul>
</li>
<li>
<p>Global Cache (Redis)</p>
<ul>
<li>분산 환경에서 데이터 공유를 위한 캐시 계층으로 DB 조회 분산 및 캐시 히트율 확보</li>
<li>MongoDB Atlas Search의 검색 연산 부하 분산</li>
</ul>
</li>
<li>
<p>캐시 정책: 데이터 신선도 유지를 위한 L1(Caffeine, 1m) 캐시와 적중률 향상을 위한 L2(Redis, 10m) 캐시의 다중 계층화로 검색 시스템 성능 최적화</p>
</li>
</ul>

<li>

</li>

<h4 id="4-1-no-offset-기반-mongodb-대용량-배치-처리">4-1. No-offset 기반 MongoDB 대용량 배치 처리</h4>
<h5 id="문제-상황-3">문제 상황</h5>
<ul>
<li>MongoDB 커서 기반 스트리밍 방식 적용 시, 배치 장애 발생 지점 기록 및 재시작 처리에 한계가 존재</li>
<li>일반적인 페이징 방식에서는 페이지가 깊어질수록 Skip 연산으로 인해 <span class="katex"><span class="katex-mathml"><math xmlns="http://www.w3.org/1998/Math/MathML"><semantics><mrow><mi>O</mi><mo stretchy="false">(</mo><mi>n</mi><mo stretchy="false">)</mo></mrow><annotation encoding="application/x-tex">O(n)</annotation></semantics></math></span><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:1em;vertical-align:-0.25em;"></span><span class="mord mathnormal" style="margin-right:0.02778em;">O</span><span class="mopen">(</span><span class="mord mathnormal">n</span><span class="mclose">)</span></span></span></span> 스캔 비용이 발생</li>
</ul>
<h5 id="개선-내용-3">개선 내용</h5>
<ul>
<li>
<p>Spring Batch <code>ExecutionContext</code>에 마지막 처리 ID를 저장하는 No-offset 기반 <code>MongoPagingItemReader</code> 구현</p>
</li>
<li>
<p><code>ItemStream</code> 구현을 통해 인덱스 기반의 포인터 상태를 직접 관리하도록 설계</p>
</li>
</ul>
<h5 id="결과-3">결과</h5>
<ul>
<li><strong>성능 일관성 확보</strong>: 페이징 오버헤드를 제거하여 데이터 양에 관계없이 일정한 쿼리 성능을 유지</li>
<li><strong>운영 안정성 구축</strong>: 장애 시 메타데이터를 활용하여 중단 지점부터 즉시 재개 가능한 환경을 마련</li>

<h4 id="4-2-llm-기반-비정형-데이터-구조화-및-배치-처리-최적화">4-2. LLM 기반 비정형 데이터 구조화 및 배치 처리 최적화</h4>
<h5 id="문제-상황-4">문제 상황</h5>
<ul>
<li>비정형 크롤링 데이터는 노이즈가 많아, 정규표현식 기반 전처리만으로는 정제 정확도와 확장성을 확보하는 데 한계가 존재</li>
<li>수천 건의 데이터를 데이터별로 개별 LLM API 호출 방식으로 처리하면, 과도한 RTT와 누적 호출 비용으로 인해 성능 및 비용 병목이 발생</li>
</ul>
<h5 id="개선-내용-4">개선 내용</h5>
<ul>
<li><strong>Batch Prompting 최적화</strong>: 단일 요청에 데이터를 N건씩 묶어 처리하여 API 호출 횟수를 축소하고 네트워크 RTT 오버헤드 극복</li>
<li>LLM 반환 JSON 텍스트 정제 및 <code>ObjectMapper</code>를 활용한 자바 객체 역직렬화 적용</li>
<li><strong>장애 격리 처리</strong>: 일부 데이터 처리 실패 시 전체 배치가 중단되지 않도록, 실패 데이터를 별도 DB에 저장해 분리 관리하도록 구현</li>
</ul>
<h5 id="결과-4">결과</h5>
<ul>
<li>
<p><strong>비용 및 속도 최적화</strong>: 1:1 API 호출 대비 네트워크 비용을 50% 절감하고 전체 배치 지연 시간을 단축</p>
</li>
<li>
<p><strong>데이터 품질 확보</strong>: 불필요한 노이즈를 제거하여 서비스에 즉시 활용할 수 있는 정형 데이터셋을 구축</p>
</li>
</ul>
